# 订单类应用接入方案

### 目标应用分类

打单、进销存、订单处理。

### 应用入云要求

应用必须部署到多多云内，多多云相关入云要求和操作参考[入云指南](https://open.pinduoduo.com/application/document/browse?idStr=8C688AAF027C41B9)。

### 接入规范要求

（1）订单敏感数据加密存储至DB、ES等介质中，存储层不允许出现明文

（2）浏览器界面默认展示打码信息，如 张**，18*******07

（3）浏览器界面展示明文可通过 oplb 直接解密

（4）Oplb解密明文接口必须满足规范要求：

/pdd/control/decrypt/v1/receiverPhone

/pdd/control/decrypt/v1/receiverName

/pdd/control/decrypt/v1/receiverAddress

/pdd/control/decrypt/v1/cardnum

/pdd/control/decrypt/v1/cardname

浏览器端发起的其他http接口请求不允许出现敏感明文信息

（5）云内外发的订单敏感数据明文后，需通过SDK上报至平台

（6）解密接口默认只能云内调用，并且解密接口做店铺级别调用限流改造验收标准

（7）不允许出网 tcp 访问，已开通的tcp需改造后下线

### 业务改造方案

涉及敏感信息的业务场景主要有如下几个：

- 前端订单展示与查询
- 短信
- 电子面单获取
- 订单合单业务

这几个业务场景分别对应的改造方案如下。

#### 前端订单展示与查询

**订单列表打码**

1）应用内订单列表及订单详情中，默认展示的敏感字段信息均为掩码模式（俗称"打码展示"），各类型字段掩码模式如下掩码标准给出

2）应用内敏感字段旁新增查看按钮，用户只有在点击完成后才允许明文可见数据，按钮文案应用自行定义（注：不允许实现一键查看全部订单信息等批量查看功能，仅允许单次查看敏感字段明文，即字段级展示）

涉及敏感信息字段及其对应改造后掩码如下：

| 敏感字段 | 描述 | 打码要求 | 打码示例 |
| --- | --- | --- | --- |
| receiver_name | 收件人姓名 | 保留收件人第1位或首尾2个汉字，中间以"*"拼接 | 牧*、牧*川 |
| receiver_phone | 收件人电话 | 保留手机号首尾各2位 | 18*******33 |
| receiver_address | 收件人地址，不拼接省市区 | 替换含阿拉伯数字的字符为"*" | 上海市上海市长宁区娄山关路***号*栋 |
| address | 收件详细地址 | 替换含阿拉伯数字的字符为"*" | 娄山关路***号*栋 |

**订单敏感数据展示改造**

在打码方案中讲述一条规则“不允许实现一键查看全部订单信息等批量查看功能，仅允许单次查看敏感字段明文，即字段级展示”。所以不是简单获取订单明文，改造方案需要符合以后数据安全的策略，避免后期重复修改业务代码。几个要点：

1、 浏览器获取明文的http服务接口实现必须按照规范来实现

2、 出入参遵循规范

3、 解密是SLB网关来解密，应用接口直接返回密文即可，如果不是规范返回会解密失败

**浏览器获取明文的http服务接口实现必须按照规范来实现**

如上文所述，终端获取明文的调用请求，请求的路径必须严格遵循 /pdd/control/decrypt/v1/xxx 的格式。共有 6 个加密数据可以被解密，每一个数据的调用路径如下：

•	/pdd/control/decrypt/v1/receiverPhone （获取收货人手机号）

•	/pdd/control/decrypt/v1/receiverName （获取收货人姓名）

•	/pdd/control/decrypt/v1/receiverAddress （获取收货人地址）

•	/pdd/control/decrypt/v1/cardnum （获取收货人身份证号）

•	/pdd/control/decrypt/v1/cardname （获取收货人身份证姓名）

•	/pdd/control/decrypt/v1/address（获取收货人的详细地址）

**出入参遵循规范**

获取明文数据的调用中的请求参数和返回参数要严格遵循规范，否则平台网关无法顺利解密。具体规范如下：

请求参数：

| 请求协议 | POST |  |
| --- | --- | --- |
| 参数规范 | { "decrypt_set": { "sub_msg": "", "sub_code": "" }, "request_id": "", "page_table_id": "", "order_sn": "" } |  |
| 参数释义 | sub_msg | 可选参数。 如果解密失败，需要返回给到前端系统的提示信息。 |
|  | sub_code | 可选参数。 如果解密失败，需要返回给到前端系统的提示代码。 |
|  | request_id | 可选参数。 用于前端传入请求号，如果系统有自己的解密请求管理，可以主动传入这个号，返回参数中的 request_id 会填充传入的这个参数值；否则，平台会自动生成一个 request_id 作为返回参数。 |
|  | page_table_id | 可选参数。 前端展示列表的 id，用于标识解密的数据属于前端展示列表的哪个位置，这样在请求返回时，可以根据这个值将明文填充到对应的表格位置。 |
|  | order_sn | 必选参数。 需要解密的数据所属的订单编号。 |

返回参数：

|  |  |  |
| --- | --- | --- |
| 参数规范 | { "page_table_id": "", "request_id": "", "client_id": "", "mall_id": 0, "order_sn": "", "order_info": { "xxx" : "" }, "error_code": "", "error_msg": "", "sub_msg": "", "sub_code": "" } |  |
| 参数释义 | page_table_id | 前端展示列表的 id，用于标识解密的数据属于前端展示列表的哪个位置。 |
|  | request_id | 请求 ID。如果用户主动传入，则返回用户传入的参数值；否则，平台会自动生成一个请求 ID。 |
|  | client_id | **必选**参数，应用 ID。 |
|  | mall_id | **必选**参数，店铺 ID。 |
|  | order_sn | **必选**参数，订单编号。 |
|  | xxx | 这里的 xxx 与请求 URL 中的 xxx 有对应关系，即请求解密什么参数，即返回什么参数。映射关系如下： /pdd/control/decrypt/v1/receiverPhone -> receiver_phone /pdd/control/decrypt/v1/receiverName -> receiver_name /pdd/control/decrypt/v1/receiverAddress -> receiver_address /pdd/control/decrypt/v1/cardnum -> id_card_num /pdd/control/decrypt/v1/cardname -> id_card_name /pdd/control/decrypt/v1/address -> address 举个例子，如果前端接口请求的是 /pdd/control/decrypt/v1/receiverPhone 路径，也就是解密收货人手机号，则返回参数中 order_info 里的这个 xxx 参数名就是 receiver_phone。 |
|  | error_code | 平台错误码。如果平台系统或者请求参数有问题，导致请求失败，则会返回这个错误码。 |
|  | error_msg | 平台错误信息。如果平台系统或者请求参数有问题，导致请求失败，则会返回具体的错误信息。 |
|  | sub_msg | 解密失败错误信息。用户传入，如果解密失败，需要返回给到前端系统的提示信息。 |
|  | sub_code | 解密失败错误码。用户传入，如果解密失败，需要返回给到前端系统的提示代码。 |

**应用响应返回密文**

被调用的程序接口要符合以下几点：

**1.	接收指定调用路径的请求**

**2.	获取密文数据后，不再通过解密接口解密**

**3.	遵循请求参数和返回参数的规范进行调用和返回**

**示例请求**

因为收件人信息需要做完检索条件到存储端作查询条件。根据上述《加密数据检索方案》中建议：完整加密串 和 检索串 要分两个字段存储。如何解析分离出检索串请，请查看加密数据算法。

#### 订单合单业务改造

合单业务一般根据：收件人姓名、电话号码、收件详细地址 做合并，但这些信息都是敏感信息存储都会被加密。此时需要使用此些字段做合并逻辑需要做一些改造。

先看看加密后的完全密文结构，文本字符串类（姓名、地址）构造：

| SEP | EncryptedData | SEP | Index | SEP | Version | SEP | SEP |
| --- | --- | --- | --- | --- | --- | --- | --- |

手机号和身份证类构造：

| SEP | Index | SEP | EncryptedData | SEP | Version | SEP | SEP |
| --- | --- | --- | --- | --- | --- | --- | --- |

其中 Index 是检索信息，同一份数据多次加密后的结果都不会变。

合并业务存储方案：完整加密串 和 检索串 要分开存储，完整加密串和检索串分别存储数据表两个字段。因为加密后的数据长度会扩大10-20倍，应该减少索引字段的长度，提高检索效率。加密原串用于开展业务，检索串用于数据查询使用。

假设订单数据表order_info模型和数据如下：

| 订单号 | 手机加密原串 | 手机检索串 | 姓名加密原串 | 姓名检索串 |
| --- | --- | --- | --- | --- |
| 2020-12 | XXXXXX1 | $7AnwZJ1e6BZc$ | SSSSSS1 | 5wk1O4mgVI |
| 2020-13 | XXXXXX2 | $7AnwZJ1e6BZc$ | SSSSSSS2 | 5wk1O4mgVI |
| 2020-14 | XXXXXX3 | $aWnsqAJ1eSZc$ | SSSSSSS2 | NXiZMm/QCu |

可以通过手机检索串和姓名检索串来字符串匹配，如果字符串匹配过于慢可以加一列hash值列，如下：

![图片](https://t16img.yangkeduo.com/perse-open-api/2020-09-30/47da9ecd-0423-42e4-b1a6-5c88374a2917.png)

假设我们Index有万分之一的冲突，三个字段同时冲突的概率是 万亿分之一的概率。
