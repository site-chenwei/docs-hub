# 前端解密规范

### 基本规范

应用前端提供给商家点击解密的功能，必须要严格按照如下规范要求来实现，其中主要注意点如下：

1、默认展示给商家的是打码之后的数据，打码可使用 [脱敏接口](https://open.pinduoduo.com/application/document/api?id=pdd.open.decrypt.mask.batch) 实现

2、终端获取明文的 http 服务接口路径必须按照指定的路径规范来实现

3、出入参遵循规范

4、解密是 SLB 网关来解密，应用接口直接返回密文即可，如果不是规范返回会解密失败

5、只允许单个字段解密

### 示例程序

### **接口路径规范**

如上文所述，开发者需要自己实现前端查看解密的接口。前端获取明文的调用请求，请求的路径必须严格遵循 /pdd/control/decrypt/v1/xxx 的格式。共有 6 个加密数据可以被解密，每一个数据的调用路径如下：

•	<your_domain>/pdd/control/decrypt/v1/receiverPhone （获取收货人手机号）

•	<your_domain>/pdd/control/decrypt/v1/receiverName （获取收货人姓名）
•	<your_domain>/pdd/control/decrypt/v1/receiverAddress （获取收货人地址）

•	<your_domain>/pdd/control/decrypt/v1/cardnum （获取收货人身份证号）

•	<your_domain>/pdd/control/decrypt/v1/cardname （获取收货人身份证姓名）

•	<your_domain>/pdd/control/decrypt/v1/address（获取收货人的详细地址）

注：<your_domain> 指的是开发者云内应用的域名，比如开发者在多多云上对外提供服务的域名是 service.pavelzhang.top，则解密收件人手机号字段的解密请求就应该调用 service.pavelzhang.top/pdd/control/decrypt/v1/receiverPhone。

这里 /pdd/control/decrypt/v1/receiverPhone 必须直接跟在域名后面，不可以再有其他的子路径（比如 /pdd/control/decrypt/v1/receiverPhone.php 或者 pdd/control/decrypt/v1/receiverPhone/ 都是不可以的）。

### **出入参遵循规范**

获取明文数据的调用中的请求参数和返回参数要严格遵循规范，否则平台网关无法顺利解密。具体规范如下：

#### **请求参数**

| 请求协议 | POST |  |
| --- | --- | --- |
| 参数规范 | { "decrypt_set": { "sub_msg": "", "sub_code": "" }, "decrypt_report_type": 1, "request_id": "", "page_table_id": "", "order_sn": "" } |  |
| 参数释义 | sub_msg | 可选参数。 如果解密失败，需要返回给到前端系统的提示信息。 |
|  | sub_code | 可选参数。 如果解密失败，需要返回给到前端系统的提示代码。 |
|  | decrypt_report_type | 可选参数。 用于商家报备解密场景。 可取值范围 : 0, 1, 2, 3；默认值为 0。 0：默认的方式，解密获取虚拟号, 手机号是虚拟号，地址末尾加4位分机号 1：使用非拼电子面单发货，解密获取原始明文，手机号是明文，地址是原始明文 2：需要第三方代发，解密获取原始明文，手机号是明文，地址是原始明文 3：需要线下手动发货，解密获取原始明文，手机号是明文，地址是原始明文 |
|  | request_id | 可选参数。 用于前端传入请求号，如果系统有自己的解密请求管理，可以主动传入这个号，返回参数中的 request_id 会填充传入的这个参数值；否则，平台会自动生成一个 request_id 作为返回参数。 |
|  | page_table_id | 可选参数。 前端展示列表的 id，用于标识解密的数据属于前端展示列表的哪个位置，这样在请求返回时，可以根据这个值将明文填充到对应的表格位置。 |
|  | order_sn | 必选参数。 需要解密的数据所属的订单编号。 |

#### 返回参数

|  |  |  |
| --- | --- | --- |
| 参数规范 | { "page_table_id": "", "request_id": "", "client_id": "", "mall_id": 0, "order_sn": "", "decrypt_report_type": 1, "order_info": { "xxx" : "" }, "error_code": "", "error_msg": "", "sub_msg": "", "sub_code": "" } |  |
| 参数释义 | page_table_id | 前端展示列表的 id，用于标识解密的数据属于前端展示列表的哪个位置。 |
|  | request_id | 请求 ID。如果用户主动传入，则返回用户传入的参数值；否则，平台会自动生成一个请求 ID。 |
|  | client_id | **必选**参数，应用 ID。 |
|  | mall_id | **必选**参数，店铺 ID。 |
|  | order_sn | **必选**参数，订单编号。 |
|  | decrypt_report_type | 可选参数。 用于商家报备解密场景，如果要做报备，则返回参数中必须带上这个参数。 可取值范围 : 0, 1, 2, 3；默认值为 0。 0：默认的方式，解密获取虚拟号, 手机号是虚拟号，地址末尾加4位分机号 1：使用非拼电子面单发货，解密获取原始明文，手机号是明文，地址是原始明文 2：需要第三方代发，解密获取原始明文，手机号是明文，地址是原始明文 3：需要线下手动发货，解密获取原始明文，手机号是明文，地址是原始明文 |
|  | xxx | 这里的 xxx 与请求 URL 中的 xxx 有对应关系，即请求解密什么参数，即返回什么参数。映射关系如下： /pdd/control/decrypt/v1/receiverPhone -> receiver_phone /pdd/control/decrypt/v1/receiverName -> receiver_name /pdd/control/decrypt/v1/receiverAddress -> receiver_address /pdd/control/decrypt/v1/cardnum -> id_card_num /pdd/control/decrypt/v1/cardname -> id_card_name /pdd/control/decrypt/v1/address -> address 举个例子，如果前端接口请求的是 /pdd/control/decrypt/v1/receiverPhone 路径，也就是解密收货人手机号，则返回参数中 order_info 里的这个 xxx 参数名就是 receiver_phone。 **注：** **如果是针对 receiver_phone 的解密，返回结果中会额外有两个字段：** ** virtual_number_type：虚拟号类型：0-非虚拟号 1-虚拟号** ** virtual_identify_number：虚拟号分机号，****若虚拟号类型为 1，则返回该字段；若为 0，则不返回** |
|  | error_code | 平台错误码。如果平台系统或者请求参数有问题，导致请求失败，则会返回这个错误码。 |
|  | error_msg | 平台错误信息。如果平台系统或者请求参数有问题，导致请求失败，则会返回具体的错误信息。 |
|  | sub_msg | 解密失败错误信息。用户传入，如果解密失败，需要返回给到前端系统的提示信息。 |
|  | sub_code | 解密失败错误码。用户传入，如果解密失败，需要返回给到前端系统的提示代码。 |

### **应用响应返回密文**

被调用的程序接口要符合以下几点：

- **接收指定调用路径的请求**

- **获取密文数据后，不再通过解密接口解密**

- **遵循请求参数和返回参数的规范进行调用和返回**

#### **示例请求**

以电话号码为例，http 接口示例:

| url | /pdd/control/decrypt/v1/receiverPhone |
| --- | --- |
| 请求 | { "decrypt_set": { "sub_msg": "", "sub_code": "" }, "decrypt_report_type": 1, "request_id": "", "page_table_id": "", "order_sn": "" } |
| 响应 | { "page_table_id": "", "request_id": "", "client_id": "", "mall_id": 0, "order_sn": "", "decrypt_report_type": 1, "order_info": { "receiver_phone": "", "virtual_identify_number": "3299", "virtual_number_type": 1 }, "error_code": "", "error_msg": "", "sub_msg": "", "sub_code": "" } |
