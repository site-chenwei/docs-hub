# 签名与验签机制说明

更新时间：2026-02-24 19:59:48

## 

一、核心目的

签名/验签机制用于确保HTTP请求的完整性（未被第三方篡改）和来源可信性（确认为约定方发送）。

**实现逻辑：**通信双方（客户端与方舟网关、方舟网关与服务端）基于统一算法和专属密钥（secret），对请求核心内容独立生成签名，通过比对签名一致性判定请求合法性。

## 

二、流程介绍

![](https://t16img.yangkeduo.com/perse-open-api/2025-10-19/264c9340-f5ec-4a85-9965-e6ecf17f8fb5.png)

### 

1、客户端加签

-   客户端通过SDK，使用\`client\_id\`及预设算法对请求内容生成签名，随请求一同发送。
    

### 

2、方舟网关处理

-   第一步：验证客户端传入的签名合法性；
    
-   第二步：对\`target\_client\_id \`（目标服务标识）重新加签，确保服务端可验证网关身份；
    

### 

3、服务端验签

-   对target\_client\_id进行验签，从而确认请求来自合法的方舟网关，保证服务端接收请求的安全性与合法性。
    

## 

三、客户端加签算法规范

### 

（一）算法适用场景

1.  若使用官方提供的 SDK 及内置 httpClient，无需手动实现加签逻辑（SDK 已封装完整算法）；
    
2.  若自行实现 HTTP 请求调用（自定义 http call），需严格按以下规则实现加签。
    

### 

（二）支持算法

当前仅支持 算法（sign\_method=md5），加签步骤如下：

#### 

步骤 1：参数排序

收集本次请求中所有参数（含公共参数与业务参数），按参数名（key）的 ASCII 码升序排列：

-   优先比较首字符 ASCII 值，值越小越靠前；
    
-   首字符相同时，按后续字符依次比较（左到右），直至区分顺序。
    

#### 

步骤 2：字符串拼接

-   按排序后的「key+value」顺序无缝拼接（无分隔符、无空格、无换行），形成基础字符串；
    
-   在基础字符串的**头部和尾部**，分别拼接专属 client\_secret（格式：client\_secret + 基础字符串 + client\_secret），生成最终签名字符串。
    

#### 

步骤 3：生成 sign 值

对最终签名字符串执行 MD5 加密，将加密后的密文转为大写，即为最终的 sign 值。

四、服务端验签流程

### 

（一）验签步骤

服务端需自行编写代码，完成 target\_client\_id 等信息及密钥的校验工作。为确保接口调用方为方舟网关，服务端需实现验签逻辑以验证请求合法性。

开发者需按以下验签算法执行校验流程：

1\. 从请求头（Header）中提取签名信息；

2\. 使用以下验签算法完成验签。

### 

（二）验签代码示例（Java）

 \*

    headers,String body,String secret){

## 

五、注意事项

### 

（一）密钥安全规范

1、secret 是签名机制的核心，需严格保密，禁止硬编码在客户端代码、前端页面、配置文件等可被外部获取的位置；

2、建议通过安全密钥管理系统（如 KMS）存储和获取 secret，定期轮换密钥（如每 3 个月）；

3、若怀疑 secret 泄露，需立即重置，并同步更新相关通信方的密钥配置。

### 

（二）参数编码规则

1、所有请求 / 响应的数据编码统一为 UTF-8，禁止混用其他编码格式；

2、URL 中的所有参数值，必须做 URL-Encode 编码；

3、若请求头 Content-Type 为 ：所有参数值需做 URL-Encode 编码；

4、若请求头 Content-Type 为 ：每个表单字段的参数值无需编码，但需在表单字段中指定 charset=utf-8；

5、若请求为 JSON 格式（Content-Type: application/json），请求体无需 URL-Encode，但需确保 JSON 字符串格式规范（无多余空格、换行）。

### 

（三）异常处理规范

1、验签失败时，服务端需立即拒绝处理请求，返回明确错误信息（如「签名无效」「pdd\_sign 字段缺失」），禁止返回业务相关敏感信息；

2、建议记录验签失败日志（含请求时间、请求来源 IP、错误类型），便于排查异常（如密钥不匹配、参数篡改、编码不一致等）；

3、客户端若收到「签名无效」错误，需优先检查：secret 是否正确、参数排序是否符合 ASCII 规则、数据编码是否为 UTF-8、请求体是否与加签时一致。
